Overview

What is MSVA?

MSVA is a Hinglish inbound-support voice agent for the Madhusudan dairy brand. Callers dial in about milk, ghee, paneer, dahi and other SKUs; the agent answers instantly in Hinglish, resolves routine requests, takes complaints, and warm-transfers to a human only when the case truly needs it.

The same "brain" can be reached three ways: a real phone call (Exotel/Twilio), an in-browser live call you can run on a laptop, and a text demo. A dashboard turns call data into operational KPIs.

Talk

Real voice, both ways

Speak naturally; the agent transcribes, thinks, and replies in a natural Hinglish voice. Hands-free and interruptible — just like a phone call.

Act

It does real things

The agent calls tools — order lookup, inventory check, ticket creation, human transfer — and grounds its reply in the results.

Measure

Observable

Per-turn latency (ASR / LLM / TTS / time-to-first-word) is shown live; call data rolls up into a KPI dashboard.

Safe

Demo failsafe

A pre-recorded clip can be played as a native agent turn if the live pipeline misbehaves during a demo.

Foundations

Key concepts & glossary

A voice agent is a loop: turn speech into text, decide what to say and do, turn text back into speech — fast enough to feel like a conversation. These are the moving parts.

TermWhat it means here
ASR (speech-to-text)Converts the caller's audio into text. MSVA uses Sarvam (saaras), tuned for Indian languages and Hinglish.
TTS (text-to-speech)Converts the agent's text reply into audio. MSVA uses Sarvam bulbul:v3, streaming raw PCM so playback starts before the whole sentence is synthesized.
VAD / endpointing"Voice Activity Detection" — deciding when the caller has stopped talking so the agent can respond. MSVA ships a simple energy/RMS endpointer (swap to Silero for production).
Barge-inLetting the caller interrupt the agent mid-sentence. When the caller starts speaking over the bot, the agent stops talking and listens.
LLM + tool callingThe language model (local Ollama) writes the reply and decides when to call a tool (a function like lookup_order). Results are fed back so the reply is grounded in real data.
SSEServer-Sent Events — how the agent streams its reply token-by-token to whoever is listening (the browser or the telephony service).
TransportThe "pipe" carrying call audio. MSVA has two: an Exotel/Twilio phone transport and a browser-WebSocket transport. Both feed the same pipeline.
Time-to-first-word (TTFW)The headline latency metric: how long from the caller finishing their sentence to hearing the first word of the reply.
FallbackDeterministic, scripted Hinglish replies (+ heuristic tool calls) used whenever the LLM is unavailable or too slow — so the demo never goes silent.

Big picture

System architecture

MSVA is a small monorepo of three services plus shared types. The web app talks to the API for analytics/chat and to the telephony service for live-call audio. The telephony service is the orchestrator that bridges call audio to the agent brain.

PackagePortResponsibility
apps/web5173React/Vite UI: dashboard, Live Call, Voice Playground, VA Demo, this guide.
apps/api4100The agent brain (streamChat), analytics from CSV, Sarvam TTS proxy, tools/backends, demo failsafe.
apps/telephony4200Per-call pipeline: ASR, VAD, TTS, barge-in. Two transports — browser WS & Exotel.
packages/shared—TypeScript types shared across all apps (call state, stream events, metrics).

User guide

Using the app

The web app has a left sidebar with five areas. Most people will spend their time in Live Call.

  • Call Analytics — KPI dashboard built from historical call data.
  • Live Call — have a real spoken conversation with the agent in your browser.
  • VA Demo — type caller lines and read/hear the agent's replies (no mic).
  • Voice Playground — audition Sarvam voices on any phrase.
  • Platform Flow — a visual summary of how a call is handled.

Live Call — talking to the agent

This is the headline experience: press Call, speak in Hindi/Hinglish, and the agent talks back. It's hands-free and you can interrupt it.

  1. Open Live Call, pick a caller profile (the persona you're role-playing) and an agent voice.
  2. Press Call and allow microphone access.
  3. After the greeting, just start talking. Pause, and the agent replies.
  4. To interrupt, simply talk over it — it stops and listens (barge-in).
  5. Use Mute to hold your mic, End to hang up.

Use headphones. Without them, the agent's voice from your speakers can be picked up by the mic and make it interrupt itself. Echo cancellation helps, but headphones are foolproof.

The mic only works over HTTPS (or localhost). On the deployed site that's https://msva.maplestudios.co.in.

Demo failsafe

If the live pipeline misbehaves during an important demo, you can fall back to a pre-recorded clip that plays as a native agent turn — the speaking indicator, a caption that types out in sync with the audio, and a latency/outcome readout all appear exactly as in a real reply.

  • A subtle Demo failsafe button sits under the call controls.
  • If a call fails to start, it becomes a prominent Play demo response instead.
  • The clip is served first-party by the API, so it looks like the tool generated it.

The on-screen caption text comes from data/demo-failsafe.json (transcript). Set it to the exact words of your clip so the caption matches.

Reading the latency panel

During a live call the right panel shows real numbers for the most recent turn, plus a rolling average for time-to-first-word.

MetricMeaning
Time to first wordCaller stops talking → first audio of the reply is heard. The number that "feels" like responsiveness.
ASRHow long transcription took after end-of-speech.
LLM first tokenTime until the model produced its first word.
LLM totalTime for the full reply to generate.
TTS first byteTime until the first chunk of synthesized audio was ready.

Dashboard · VA Demo · Voices

Call Analytics summarizes inbound demand: total/unique callers, human answer rate, unserved demand (missed + unanswered), repeat-caller load, time-band coverage, and agent load — the case for AI coverage.

VA Demo is the no-mic path: type a caller line, read the reply, and tap the speaker icon to hear it. Good for scripted walk-throughs.

Voice Playground lets you type any phrase and audition it across Sarvam voices to pick a permanent agent voice.

It's a pnpm workspace. Node ≥ 22, pnpm 10.

Running locally

pnpm install
pnpm dev:all      # api(4100) + web(5173) + telephony(4200)

Then open http://localhost:5173. Real voice needs SARVAM_API_KEY in apps/api/.env and apps/telephony/.env. Natural replies (optional) need Ollama running locally; without it the deterministic fallback keeps everything working.

The voice pipeline (in-browser call)

The browser is treated as one more telephony transport. Mic audio is downsampled in an AudioWorklet to 16 kHz PCM and streamed over a WebSocket to /browser; the telephony service runs ASR, detects end-of-speech, asks the agent, and streams TTS audio back for gapless playback.

Key files: apps/web/src/callClient.ts (capture + playback + WS), apps/telephony/src/browserPipeline.ts (orchestration), asr/sarvam.ts, tts/sarvam.ts, vad.ts, agentClient.ts (SSE to the API).

Function calling (two-pass)

The agent doesn't fake tool calls. streamChat runs a decision pass with tool schemas; if the model chooses a tool, it's executed against a real backend, the result is appended to the thread, and a second streaming pass voices a reply grounded in that result. If the LLM is off or unreachable, deterministic fallback takes over.

Tools live in apps/api/src/tools/. lookup_order is a real backend that queries data/orders.json by order/invoice number or caller phone; the rest are stubs ready to wire to Freshdesk/Zoho/WhatsApp. Set AGENT_LLM=off to skip the model entirely for an instant demo.

Real phone path (Exotel / Twilio)

The phone transport mirrors the browser one. On an inbound call, the carrier POSTs to /exotel/incoming; the service returns markup telling the carrier to open a media-stream WebSocket to /voice, then the same ASR → agent → TTS loop runs.

  • Exotel: returns ExoML <Stream>; audio is 8 kHz linear PCM.
  • Twilio: return TwiML <Connect><Stream>; audio is 8 kHz µ-law (add a µ-law↔PCM transcode).

Expose the service publicly (ngrok in dev, Caddy in prod) and point the carrier's voicebot flow at it. Full steps: docs/live-call.md.

Latency metrics

The browser pipeline timestamps each turn — end-of-speech, first token, first audio byte, completion — and emits a LiveTurnMetrics control message (defined in packages/shared). The call screen renders the latest turn plus a rolling TTFW average. The CallTurn type captures the same breakdown for real phone calls.

Extending

  • Swap ASR/TTS: implement the StreamingAsr/StreamingTts interfaces in apps/telephony/src/asr|tts and point the pipeline at your factory. The loop never changes.
  • Add a tool: add it to ToolName (shared), a handler in tools/, register it in tools/index.ts, and add its schema to TOOL_SCHEMAS in voiceAgent.ts.
  • Better VAD: replace the energy endpointer in vad.ts with Silero (onnxruntime) — same interface.
  • Persist state: conversation state is in-memory per socket; move it to Redis keyed on the call id for restart-safety.

Deployment

Production runs on a single Ubuntu VPS (Hostinger KVM2). The two backend services run under pm2, the web build is served as static files, and a reverse proxy terminates HTTPS and routes traffic. The kit ships a Caddy config and a one-shot deploy/bootstrap.sh, but on a box that already hosts other sites you integrate with the existing web server — the live deployment uses nginx (deploy/nginx-msva.conf) behind Cloudflare, with the cert issued by certbot.

The proxy routes /api/* → 4100, the /browser WebSocket and the Exotel /voice stream → 4200, and serves the static build (apps/web/dist) for everything else. Full runbook: DEPLOY.md; real phone setup: docs/exotel-inbound.md.

Operational gotchas (learned in production)

Start pm2 from the repo root via the ecosystem file. Each service loads its .env from its own folder, so it must launch with cwd = apps/api / apps/telephony (which deploy/ecosystem.config.cjs sets). Start it in the wrong directory and it loads no env — the symptoms are stub ASR (every caption reads "haan mujhe paneer…") and templated fallback replies. Verify with pm2 info msva-api | grep cwd.

The Sarvam key must be in BOTH apps/api/.env (TTS preview) and apps/telephony/.env (live-call ASR + TTS). A missing telephony key gives the canned transcript above.

The proxy must serve only apps/web/dist, never the repo root — otherwise source files and .env secrets become publicly downloadable.

Real LLM replies + order lookup. On this 2‑vCPU VPS a local 3B model is too slow for live voice (~30s/turn), so the recommended brain is hosted Claude: set LLM_PROVIDER=anthropic, AGENT_LLM=on, and ANTHROPIC_API_KEY — fast and smart. The local option is LLM_PROVIDER=ollama + ollama pull llama3.2:3b (free but slow). Either way, AGENT_LLM=off (or the in‑app Instant toggle) gives instant scripted replies.

.env load order. Each service loads its .env via a first-imported env.ts (resolved relative to the file), so env vars are set before any module reads them. Don't move that import or revert to a late dotenv.config() — that previously caused the model and Sarvam key to read as empty (fallback replies + stub ASR).
